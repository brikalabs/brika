# Internationalization

A plugin's blocks, bricks, pages, and preferences can ship translations. The hub's UI uses them automatically; bricks and pages can read them with `useLocale()`.

## File layout

```
src/i18n/
  en.json
  fr.json
```

One JSON file per locale, with keys grouped by domain:

```json
{
  "devices": {
    "title": "Devices",
    "scan": "Scan for new devices",
    "online": "Online",
    "offline": "Offline"
  },
  "settings": {
    "apiKey": "API key",
    "saved": "Saved"
  }
}
```

Add `"i18n"` to your manifest's `brika` block:

```json
"brika": {
  "i18n": { "defaultLocale": "en" }
}
```

(The exact location is your `brika` namespace inside the manifest — see [Manifest Reference](manifest.md).)

## Using translations in bricks and pages

```tsx
import { useLocale } from '@brika/sdk/ui-kit/hooks';

export default function DevicesPage() {
  const { t, locale, changeLocale, formatDate, formatTime, formatNumber } = useLocale();

  return (
    <div>
      <h1>{t('devices.title')}</h1>
      <button>{t('devices.scan')}</button>
      <p>Last scan: {formatDate(Date.now())}</p>
    </div>
  );
}
```

`t(key)` looks up the value for the current locale, falling back to the default locale. `{{var}}` interpolation works:

```json
{ "devices": { "found": "Found {{count}} devices" } }
```

```tsx
t('devices.found', { count: 3 });  // "Found 3 devices"
```

`changeLocale(code)` swaps the active locale — the change propagates through every brick and page rendered in the UI.

## In the plugin process (server-side)

There is no server-side `t()` helper today. If you need to log a localised message, do the lookup in the browser side and pass the result to the server — or, for now, log in English.

## Block / brick / preference labels

The UI uses the field names from your manifest as keys. For example, a block defined as:

```json
"blocks": [
  { "id": "timer", "name": "Timer", "description": "Set a timer" }
]
```

falls back to literal English. To localise it, register translations under the convention the UI looks for:

```json
{
  "blocks": {
    "timer": {
      "name": "Minuteur",
      "description": "Démarre un minuteur"
    }
  }
}
```

The exact key conventions are documented under [i18n Pipeline](../architecture/i18n-pipeline.md) — the build tooling extracts and types them automatically.

## Build tooling

The repo includes `@brika/i18n-dev`, a Vite plugin used during development. It:

* Watches `src/i18n/*.json` for changes and hot-reloads the UI.
* Extracts `{{var}}` variables from translation strings and generates TypeScript types so `t('devices.found', { … })` is type-checked.
* Lets you open a translated string in your editor with one click from the UI (a dev-only `?translate=key` query parameter activates the overlay).

See [i18n Pipeline](../architecture/i18n-pipeline.md) for the full build story.

## Bundling locales

The `files` field in `package.json` should include both `src/i18n/` (if you ship source) and `locales/` (if you ship compiled bundles):

```json
"files": ["src", "locales", "icon.svg", "README.md"]
```
