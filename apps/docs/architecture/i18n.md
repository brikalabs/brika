# Internationalization (i18n)

Brika ships translations from three sources, all loaded through one shared runtime (`@brika/i18n`) and served from a single bulk endpoint (`GET /api/i18n/bundle/:locale`).

## The three sources

| Source            | Layout                                       | Namespace                          |
|-------------------|----------------------------------------------|------------------------------------|
| Hub               | `apps/hub/src/locales/<lang>/<ns>.json`      | bare `<ns>` (`common`, `nav`, …)   |
| Workspace package | `packages/<X>/locales/<lang>/*.json`         | package name without scope (`X`)   |
| Runtime plugin    | `<pluginDir>/locales/<lang>/*.json`          | `plugin:<packageName>`             |

Hub keeps file-per-namespace because it has many of them. Workspace packages and plugins flat-merge every `.json` file in a locale directory into a single namespace, so nested keys live inside the JSON instead of in filenames.

## Adding translations

**Hub** — create `apps/hub/src/locales/<lang>/<ns>.json`. The file becomes namespace `<ns>`. Reload happens automatically in dev via `fs.watch`.

**Workspace package** — create `packages/<X>/locales/<lang>/<anything>.json`. Boot scans `packages/*/locales/` and registers each package automatically. Namespace is `<X>` (the package name with the `@brika/` scope stripped — e.g. `@brika/permissions` becomes `permissions`). No code change anywhere else.

**Runtime plugin** — drop `locales/<lang>/*.json` in the plugin's root. The plugin manager calls `registerPluginTranslations` on install; the namespace is `plugin:<packageName>`. UI components inside a plugin shell pick this up automatically through the `PluginContext`.

## Locale fallback

`fr-CH` → `fr` → `en`. Regional variants fall back to their base language, and any missing keys ultimately fall back to English. The same chain applies to every namespace.

## Type safety

`@brika/i18n-devtools` scans every source at dev time and generates types for the keys it sees. Vite is configured (in `apps/ui/vite.config.ts`) to run the validator on JSON edits and surface missing-key / dead-key warnings in the dev overlay. Day-1, `t()` accepts any string and unknown keys produce a missing-key warning at runtime; later phases of the i18n rework tighten this into a compile error.

## Wire format

The UI fetches `GET /api/i18n/bundle/:locale` once at boot and gets every registered namespace in one response — `{ "common": {…}, "permissions": {…}, "plugin:my-plugin": {…}, … }`. Late-registered plugins (installed after boot) trigger a debounced refetch when `t()` encounters their namespace.
