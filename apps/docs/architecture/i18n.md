# Internationalization (i18n)

Brika serves translations from three sources, merged through one shared runtime (`@brika/i18n`) and exposed over a single HTTP endpoint (`GET /api/i18n/bundle/:locale`). The browser fetches once, caches against an ETag, and hydrates everything from that one response — no per-namespace round-trips.

## The three sources

| Source            | Layout                                       | Namespace                          |
|-------------------|----------------------------------------------|------------------------------------|
| Hub               | `apps/hub/src/locales/<lang>/<ns>.json`      | bare `<ns>` (`common`, `nav`, …)   |
| Workspace package | `packages/<X>/locales/<lang>/*.json`         | package name without scope (`X`)   |
| Runtime plugin    | `<pluginDir>/locales/<lang>/*.json`          | `plugin:<packageName>`             |

Hub uses file-per-namespace because it has many. Workspace packages and plugins flat-merge every `.json` file in a locale directory into a single namespace — nested keys live inside the JSON, not in filenames.

## Adding translations

**Hub** — drop a file at `apps/hub/src/locales/<lang>/<ns>.json`. The filename becomes the namespace. The hub watcher hot-reloads in dev (`fs.watch` debounced 300 ms).

**Workspace package** — drop `packages/<X>/locales/<lang>/<anything>.json`. Boot scans `packages/*/locales/`; the namespace is `<X>` with the `@brika/` scope stripped (`@brika/permissions` → `permissions`). No code change anywhere else.

**Runtime plugin** — drop `locales/<lang>/*.json` in the plugin's root. `PluginManager` calls `registerPluginTranslations()` on install and the namespace becomes `plugin:<packageName>`.

## Locale fallback

`fr-CH` → `fr` → `en`. Regional variants fall back to their base language; missing keys fall back to English. The chain is applied per-key, so a `fr-CH` namespace that's missing a single string still resolves it from `fr` or `en` automatically.

## Wire format — one endpoint

```
GET /api/i18n/bundle/:locale
→ ETag: "<content-hash>"
→ { "common": {…}, "permissions": {…}, "plugin:my-plugin": {…}, … }
```

The body is the resolved translation tree for `locale`, every registered namespace merged together. The ETag is a content-hash of the body, recomputed only when the registry mutates (the cache lives on the registry itself — see `BundleJsonCache`). Subsequent client requests send `If-None-Match: "<hash>"` and get a `304` with empty body until something on disk changes.

Two ancillary GET endpoints are unauthenticated and exist for tooling:
- `GET /api/i18n/locales` — list of available locales
- `GET /api/i18n/namespaces` — list of registered namespaces

There is **no `GET /api/i18n/:locale/:namespace`** — the bundle endpoint is the single read surface for the browser.

## Browser load lifecycle

1. **Boot.** `createI18n()` initialises i18next with the eager namespaces (default + any opt-in additions like `nav`, `auth`, `settings`). i18next's backend issues a `read(lang, ns)` per namespace; every concurrent call fans into one bulk fetch via the loader's in-flight `Map`. **One HTTP request hydrates everything.**
2. **Render.** `useTranslation('common')` reads from i18next's resource store — no network. The first render unblocks once the boot fetch resolves; subsequent renders are synchronous.
3. **Language switch.** `switchLanguage('fr')` pre-fetches the FR bundle (one request, same dedup), then calls `i18n.changeLanguage('fr')`. React renders see populated bundles instead of flashing translation keys.
4. **Late namespace** (e.g. a plugin installed after boot — see next section). One conditional bundle revalidation; either the fresh bundle has it or the `<lang>:<ns>` pair is recorded known-missing and won't re-fetch.

The loader caches per-language `{ bundle, etag }` entries indefinitely (translations almost never change at runtime). No SSE, no polling — pages load once and stay loaded.

## What happens when a plugin is installed at runtime

The browser holds the old bundle in cache. The new plugin's namespace isn't in it. Nothing reloads proactively (we removed the SSE channel that used to broadcast registry mutations — it caused excessive reload storms for vanishingly rare events).

The fallback fires the first time a component using the plugin's namespace renders:

```
useTranslation('plugin:foo')
  → i18next backend.read('en', 'plugin:foo')
  → loader.load('en', 'plugin:foo')
  → cache.get('en')['plugin:foo']  // undefined — namespace not in cached bundle
  → fetchBundle('en')               // conditional GET with If-None-Match
                                    // server: registry mutated → ETag changed → 200 + fresh bundle
  → fresh['plugin:foo']             // now defined
  → return data → component renders
```

One conditional HTTP request, one parse, one i18next update. Concurrent component mounts for different new namespaces dedupe via the in-flight `Map` — still one request total. If the bundle truly doesn't have the namespace after revalidation (typo, missing locale folder), the `<lang>:<ns>` pair is recorded known-missing and future lookups short-circuit to `{}` without re-fetching.

If a smoother UX matters for a specific flow, the install API can return the fresh bundle in its response and the UI can call `hydrateTranslations(bundle)` directly — zero round-trip, no Suspense flash.

## Hot reload during development

In dev the runtime stays HMR-agnostic. `@brika/i18n-dev`'s Vite plugin watches every locale source, scans on change, and broadcasts the full `lang → namespace → data` tree over HMR as `'i18n-dev:translations'`. The plugin's auto-injected browser entry (the same `<script>` that mounts the dev overlay) subscribes to that event and calls the runtime's public `hydrateTranslations()` seam. No SSE, no extra network, no app-side wiring.

The Vite plugin separately opens an SSE connection to `/api/i18n/events` on the hub — but that's a **Vite-server → hub** channel (so the dev validator picks up admin edits made through the hub's source-write API). The browser never opens that SSE.

```
file change (JSON edit)
  → Vite watcher
  → i18n-dev Vite plugin: scan + validate
  → server.hot.send('i18n-dev:translations', { en: {…}, fr: {…}, … })
  → browser entry: import.meta.hot.on('i18n-dev:translations', payload => hydrateTranslations(payload))
  → loader.hydrate(lang, bundle) per language → i18next.addResourceBundle()
  → React re-renders affected components
```

The event name and payload type are owned by `@brika/i18n-dev` via a `vite/client` `CustomEventMap` augmentation, so both sender (plugin) and receiver (entry script) are typechecked end-to-end. A rename of `HMR_TRANSLATIONS` in `hmr-events.ts` fails the build because the augmentation references the constant via `typeof`.

## Security defenses

Every translation payload coming off the wire goes through `sanitizeTranslationTree` from `@brika/i18n` before reaching i18next:

- **Prototype pollution** — `JSON.parse` materialises `__proto__` as an own enumerable property. A hostile or compromised server returning `{"__proto__": {...}, "common": {...}}` would otherwise reach `Object.entries()` loops and pollute `Object.prototype` through i18next's deep-merge paths. The sanitizer recursively drops `__proto__`, `constructor`, `prototype` at every depth, and the loader stores bundles as `Object.create(null)` so bracket-access can't reach the prototype chain either.
- **Public `hydrateTranslations()`** — applies the same sanitization (the runtime trust boundary is the receiver, not the producer). Unsafe language codes (e.g. `__proto__` as a language key) are filtered by `isUnsafeKeySegment` before the loader is called.
- **ETag length cap** — server-returned `ETag` header is capped at 256 chars before caching, so a hostile server can't fill the cache map with massive header strings.
- **`escapeValue: false`** — i18next escaping is off because React's JSX escaping handles output. This is safe **only** when `t()` results are rendered as React text children. Feeding `t()` into `dangerouslySetInnerHTML` or `<Trans components>` with raw HTML re-opens the XSS surface; see the inline comment in `client.ts` for the threat model.
- **Source-edit write API** (`POST /api/i18n/sources/:namespace/:locale`) is behind `requireAuth()` + `requireScope(Scope.ADMIN_ALL)` AND gated by the `BRIKA_ALLOW_I18N_EDITS=1` env flag. Disabled by default in production.

## Type safety

`@brika/i18n-dev` scans every source at dev time and generates typed `BrikaI18n.Namespaces` augmentations into `node_modules/.cache/`. Vite is configured (in `apps/ui/vite.config.ts`) to run the validator on JSON edits and surface missing-key / dead-key warnings in the dev overlay. `t()` autocomplete works against the generated types; unknown keys produce a missing-key warning at runtime today and a compile error in a later phase.

## Public runtime API

`@brika/i18n/react` exports:

| Symbol                     | Use                                                                                    |
|----------------------------|----------------------------------------------------------------------------------------|
| `createI18n(options)`      | Initialise i18next with the bulk-bundle backend. Call once at app startup.             |
| `switchLanguage(target)`   | Race-free language switch — pre-loads the target bundle before flipping `i18n.language`. |
| `hydrateTranslations(bundle)` | Push pre-fetched `lang → ns → data` into the runtime. Used by HMR bridges, SSR, tests. |
| `useTranslate()`           | Hook: `t`, `tp`, `locale`, `changeLocale`. Cheap.                                       |
| `useIntl()`                | Hook: formatters, display names, clock preference. Heavier.                            |
| `useLocale()`              | Hook: composes the two above when a component needs both.                              |

The runtime has no SSE, no polling, no dev-tooling references — `@brika/i18n` is shippable as a standalone package against any React + Vite + Bun project.
